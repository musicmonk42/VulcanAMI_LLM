# VULCAN Diagnostic Report: Memory Amnesia & Routing Timeout

## Executive Summary

After thorough analysis of the VULCAN codebase, I have diagnosed the two reported issues:

1. **Memory Amnesia (Privacy Refusals)**: NOT a code bug - the refusal messages are generated by the underlying GPT model, not VULCAN code
2. **30-Second Routing Timeout**: Working as designed - a safety mechanism to prevent indefinite hangs

---

## Issue 1: Memory Amnesia - Privacy Refusals

### Symptom Reported
```
User: "My name is Brian and I live in Alabama"
VULCAN: "your name is indeed Brian..." [processes it fine]

[30 seconds later, same conversation]

User: "what is my name?"
VULCAN: "I do not have access to personal information about users"
```

### Investigation Results

#### Search for Refusal Text
I searched the entire codebase for the reported refusal patterns:

| Search Pattern | Result |
|---------------|--------|
| `"do not have access to personal information"` | **NO MATCHES** |
| `"designed to respect user privacy"` | **NO MATCHES** |
| `"do not have the capability to retain"` | **NO MATCHES** |
| `"personal information"` (case-insensitive) | NO matches in system prompts |

**Conclusion**: The refusal text is NOT hardcoded in VULCAN.

#### System Prompt Analysis

The system prompt (`main.py:5153-5157`) does NOT instruct privacy refusals:
```python
system_prompt = (
    "You are VULCAN, an advanced AI assistant powered by specialized reasoning engines. "
    "When reasoning analysis is provided in the prompt, you MUST incorporate it directly..."
)
```

### Root Cause

The privacy refusal messages are **generated by the underlying GPT-3.5-turbo model** based on its training, NOT by VULCAN's code. 

This is a known behavior of OpenAI's models when:
1. The context contains personal information
2. Follow-up questions ask the model to recall/use that information
3. The model may generate a safety-oriented response based on its training

### Possible Mitigations (Not Bugs to Fix)

1. **Adjust System Prompt** to explicitly permit memory within conversation:
   ```python
   system_prompt = """You are VULCAN. You SHOULD remember and use information 
   shared within this conversation. When a user shares their name or personal 
   details in this session, you may refer back to them naturally."""
   ```

2. **Use GPT-4 or GPT-4-turbo** which has better context handling

3. **Increase conversation history** passed to the model (currently limited to last 5 messages in `llm_context`)

## Issue 2: 30-Second Routing Timeout

### Symptom Reported
```
[QueryRouter] Query routing timed out after 30.0s
[SLOW ROUTING DETECTED: 30587ms]
[Embedding] Timeout after 5.0s (5085ms)
Embedding cache: time=18964.06ms
```

### Root Cause Location

**File**: `src/vulcan/routing/query_router.py`

**Line 658-660** - The timeout constant:
```python
QUERY_ROUTING_TIMEOUT_SECONDS: float = (
    30.0  # 30 seconds max (increased from 20s to optimize complexity analysis)
)
```

**Line 4301-4313** - The timeout handler:
```python
async def route_query_async(...):
    try:
        plan = await asyncio.wait_for(
            loop.run_in_executor(executor, route_query, query, source, session_id),
            timeout=effective_timeout,  # 30 seconds
        )
        return plan
    except asyncio.TimeoutError:
        logger.warning(
            f"[QueryRouter] Query routing timed out after {effective_timeout}s. "
            f"Returning fallback plan for source={source}"
        )
        return _create_fallback_plan(query, source, session_id, timeout_exceeded=True)
```

### Why Routing Takes 30+ Seconds

The 30-second timeout is a **safety mechanism**, not a bug. The actual delay comes from:

1. **Embedding Model Loading** (`embedding_cache.py:94`):
   - First embedding computation can take 10-15 seconds
   - SentenceTransformer model needs to be loaded into memory

2. **Complexity Analysis** (`query_router.py:3286-3445`):
   - Multiple pattern matching operations
   - Safety validation checks
   - Adversarial detection

3. **CPU Contention**:
   - Multiple thread pool workers competing for CPU
   - NumPy/BLAS operations can monopolize threads

### Existing Optimizations (Already in Codebase)

The codebase already has extensive optimizations:

1. **Embedding Cache** (`embedding_cache.py`):
   - LRU cache for embeddings
   - Fast-path for simple queries
   - Batch eviction strategy

2. **Model Registry** (`models/model_registry.py`):
   - Singleton pattern for model loading
   - Pre-loading at startup

3. **Thread Limiting** (`main.py:17-24`):
   ```python
   os.environ["OMP_NUM_THREADS"] = "4"
   os.environ["MKL_NUM_THREADS"] = "4"
   os.environ["TORCH_NUM_THREADS"] = "4"
   ```

4. **Singleton Pre-warming** (`main.py:1026-1052`):
   ```python
   from vulcan.reasoning.singletons import prewarm_all
   prewarm_results = prewarm_all()
   ```

5. **Thread Pool Size** (`main.py:487-489`):
   ```python
   executor = concurrent.futures.ThreadPoolExecutor(max_workers=32)
   loop.set_default_executor(executor)
   ```

### Diagnosis Summary

The 30-second timeout is **working correctly** as a safety net. The underlying cause of slow routing is:

1. **Cold start**: First request after startup loads models
2. **CPU contention**: Multiple components competing for compute
3. **Memory pressure**: Large models can cause memory thrashing

### Recommendations

1. **Increase Pre-warming**: Ensure all models are loaded at startup
2. **Monitor Cache Hit Rates**: Check `get_embedding_cache_stats()`
3. **Consider GPU Acceleration**: Move embedding computation to GPU if available
4. **Reduce Timeout for Simple Queries**: Already implemented via `is_simple_query()` fast-path

Neither issue is a code bug in VULCAN:

1. **Memory Amnesia**: GPT-3.5-turbo's inherent behavior, not VULCAN code
